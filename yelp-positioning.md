# Yelp Positioning · 為什麼 yelp 存在

> 不是每個人都能清楚求救，
> 但群體的混亂本身，也是一種訊號。

---

## 1. yelp 不是什麼

寫在最前面，避免被誤解：

- **不是 119 的替代品。** 119 該打的時候還是要打。
- **不是另一個 AI 求救 app。** 重點不在 AI，在「能被接住的訊號」。
- **不是情緒分析器。** panic 不是要被分類的情緒，是要被當成資料的訊號。
- **不是「更好的報警電話」。** yelp 跟電話不在同一個維度。

如果 yelp 跟 119 競爭，它就會變成「多此一舉的 AI app」。

yelp 從來不在那個位置。

---

## 2. yelp 是什麼

三句話：

- **119 的資訊增強層。** 在 119 之外、之前、之後，繼續收集那些通話收不到的訊號。
- **群體危機訊號感測網。** 一個人按 yelp 是弱訊號，一百個人同時按是災情輪廓。
- **極低負擔的危機訊號捕捉。** 不要求使用者冷靜，不要求填表，不要求講清楚。

核心信念：

> **當你無法完整求救時，我們仍努力理解。**

---

## 3. 三條本質：與 119 的對照

| 軸線 | 119 | yelp |
|---|---|---|
| **訊號特性** | 強訊號、低樣本 | 弱訊號、高樣本 |
| **時序** | 同步通話 · 需通話成功 | 非同步留痕 · 按一下即可 |
| **系統位置** | 單點求救 + 人工 operator | 群體感知 + AI 前處理 |
| **資料形態** | 通話錄音 + 人工筆記 | 結構化 JSON · 可批次分析 |
| **使用門檻** | 須能說話、能講清楚、能描述位置 | 一個按鈕即可，其他訊號自動補捉 |
| **失敗模式** | 通話失敗 = 資訊歸零 | 通話失敗仍留下 panic / GPS / 語音片段 |

簡言之：

> **119 處理「有能力清楚求救」的人。**
> **yelp 處理「沒有能力清楚求救」的人。**

兩者互補，不是取代。

---

## 4. 兩句核心 slogan

> **當你無法完整求救時，我們仍努力理解。**

> **不是每個人都能清楚求救，但群體的混亂本身，也是一種訊號。**

---

## 5. 系統位置：傳統流程 vs yelp 流程

### 5.1 傳統模式 — 個人求救

```
受災者
  └─ 撥打 119
       └─ 人類 operator 接聽
            └─ 同步對話 · 一次一人
                 └─ 人工紀錄
                      └─ 人工派遣
```

問題：

- 人類 operator 有極限（同時間只能處理少量事件）
- 情緒、語氣、急迫感等訊號容易遺失
- 大量訊號無法聚合
- 完全依賴使用者的描述能力
- 通話失敗 = 整個求救鏈斷裂

### 5.2 yelp 模式 — 群體訊號

```
受災者 A、B、C、…、N
  └─ 按一下 yelp（不要求成功通話）
       ├─ raw_input (語音 / 文字 / 手勢) ─────────┐
       ├─ GPS                                  │
       ├─ 語音特徵 (語速 / 顫抖 / 重複)            │
       └─ 事件 ID                              │
                                              ▼
                              AI 語意解析（panic-signal-interpreter）
                                              │
                                              ▼
                                   結構化災情 JSON
                                              │
                  ┌───────────────────────────┼───────────────────────────┐
                  ▼                           ▼                           ▼
              即時聚合              災情熱區形成               批次 AI 排序
                  │                           │                           │
                  └───────────────────────────┴───────────────────────────┘
                                              │
                                              ▼
                                指揮中心 · 救援派遣 · 119
                              （資訊已預處理、已分流、已排序）
```

特性：

- **去中心化輸入** — 任何人都能貢獻一個訊號片段
- **非同步留痕** — 不需通話成功，按一下就生效
- **多來源聚合** — 文字 / 語音 / 手勢 / 定位 / 語音特徵
- **AI 前處理** — 把混亂訊息 → 標準 JSON，給 119 / 指揮中心當預判材料
- **群體湧現** — 100 個模糊訊號 > 1 個清楚通報

## 6. SOS 按鈕的並行動作設計

yelp 的中央按鈕**不只是按鈕**。按下後並行觸發：

1. **撥 119**（透過 `tel:` URI scheme，使用作業系統原生撥號）
2. **送 GPS 座標**（即使通話失敗也已送出）
3. **啟動背景 panic-signal 捕捉**（持續錄音 / 文字輸入 / 手勢）
4. **建立事件 ID**（給後端追蹤這次求救的所有衍生訊號）
5. **記錄非結構化訊號**（語音轉錄、語速、停頓、重複、語氣）

這代表：

> **即使通話失敗、中斷、或使用者講不清楚，
> 仍然留下可被指揮中心分析的危機痕跡。**

這是 yelp 對 119 的「資訊增強」：不取代它的核心功能（人對人對話），但補上它最脆弱的環節（訊號保全）。

## 7. 從單點到群體：Collective Sensing 情境

### 情境 A — 傳統模式

有一個人成功打 119：

> 「我在花蓮某處淹水」

很好。這是一筆強訊號。

但只有**會打**、**打得通**、**講得清**的人，才會進入 119 系統。

### 情境 B — yelp 模式

某區域同時有 100 個人按 yelp：

- panic_level 平均 0.85，明顯高於平時 baseline
- GPS 集中在某 500 公尺半徑
- raw_input 大量出現「水」、「淹」、「進來」
- 多筆 `vulnerable_flags.mobility_issue: true`（「走不了」、「擔架」、「不能動」）
- 手勢輸入出現大量平行波浪線（手繪示意水波）
- 部分手機訊號中斷 → 顯示為 `signal_lost` 的事件

**即使沒有任何一個人填完整地址或講清楚災情**，
系統也能在 60 秒內形成判斷：

> **某座標附近正在發生淹水事件，
> 規模約 100 人，
> 至少數十人為行動不便者，
> 建議優先派遣含擔架的救援隊。**

這就是：

> **Collective Intelligence**（群體智慧） ＋
> **群體語意 triangulation**（用大量模糊訊號交叉定位）

類似的概念在其他領域已經成熟：

- **Waze** — 群體交通感知（每個 GPS 點都是弱訊號，整體拼出路況）
- **地震速報** — 群體加速度感知（每個感測器都很弱，整體拼出震央）

yelp 做的是：

> **人類恐慌語意版的群體感知。**

## 8. 在公民科技與積木賽中的位置

yelp 完全契合「積木式設計」與「公民科技」的精神：

- **不是封閉 app**，而是**災情情報前處理積木**
- 後續積木都接得住這份 JSON：救援派遣、醫療支援、避難所、地圖熱區、物資調度、AI 追問、簡訊廣播、儀表板、MCP server
- 對應 **2025/09 花蓮馬太鞍溪堰塞湖**事件的真實需求 ——
  - 很多人受困
  - 訊號斷續
  - 部分人無法完整通報
  - 救災單位需要快速判斷哪裡是熱區、哪邊有行動不便者
  - 傳統 119 模式在量大時會瓶頸

yelp 不是要解決所有問題，而是：

> **把「人類崩潰時的訊號」轉成「公共系統可交換的求救資料」。**

讓既有的防災基礎建設可以接住更多原本接不到的訊號。

---

## 9. 結語

防災系統長期以來都把「使用者能清楚求救」當作預設。
這個預設對**最需要被救的人**最不公平。

yelp 重寫這個預設：

> **不要求人類冷靜，
> 而是把不冷靜本身也視為一種可分析的訊號。**

最終，yelp 想傳達的不是「請你冷靜回報」，而是：

> **就算你無法冷靜，
> 就算你說不清楚，
> 就算你只剩力氣按一下，
> 系統也會努力理解你 ——
> 而且，會把你跟周遭其他人的訊號拼成更完整的災情輪廓。**
