# Yelp UX Spec

## 1. 設計概念

Yelp 的使用者介面必須服從一個前提：人在緊急情況下沒有餘裕理解複雜介面。

因此 Yelp 不是一個讓使用者「填寫災情」的 App，而是一個讓使用者用最低負擔啟動求助資料流的手機應用程式。第一畫面只保留一個核心動作：按下中央按鈕，立即建立求助事件，送出位置，並啟動後台的災情訊號判讀。

整體 UX 的情緒不是科技感展示，而是安定、直接、少打擾。它要讓使用者在恐慌時不用思考「下一步要按哪裡」，也不用被很多資訊壓垮。

核心原則：

- 第一畫面只服務一件事：啟動求助。
- 狀態資訊預設收起，不干擾正在求生的人。
- 後台處理必須可被查看，讓使用者或陪同者知道系統正在做事。
- 每個資料項目都要有清楚狀態，例如已建立、已送出、已讀、需人工複核。
- JSON 災情情報可以預覽，因為它是 Yelp 作為防災積木的核心交付物。

## 2. 使用者情境

### 2.1 主要使用者

主要使用者是在災害或突發危機中持有手機的人。這個人可能正在逃生、受困、顫抖、無法完整說話，或正在照顧身旁的人。

此時 App 不應要求他輸入太多資訊。Yelp 要做的是讓使用者先按下按鈕，剩下的由系統主動接手。

### 2.2 次要使用者

次要使用者可能是身旁較冷靜的人、志工、家屬、現場協助者或救災人員。他們可能會打開儀表板，查看目前有哪些資訊已經送出，哪些資料仍在處理，哪些判讀結果需要確認。

因此 Yelp 需要同時照顧兩種狀態：

- 對恐慌中的人：畫面越少越好。
- 對能協助的人：狀態越清楚越好。

## 3. 主要流程

### 3.1 進入 App

使用者打開 Yelp 後，第一畫面是一個極簡手機介面：

- 中央是一個大型圓形按鈕。
- 按鈕文字建議為「Yelp」或「發出求助」。
- 畫面下方只有非常少量的狀態文字，例如「準備好建立求助事件」。
- 不顯示表單、不顯示地圖、不顯示長篇說明。

這個畫面不該像後台管理工具，而應該像緊急狀態下唯一能理解的開關。

### 3.2 按下中央按鈕

使用者按下中央按鈕後，系統立即建立一個事件。

同時執行：

- 建立事件 ID。
- 取得目前位置。
- 記錄時間。
- 啟動語音或文字訊號收集流程。
- 初始化災情 JSON。
- 將可用資料送往後端資料庫。

按鈕狀態可以改變，例如：

- 未啟動：中央按鈕穩定顯示。
- 已啟動：按鈕變成脈動狀態，表示 Yelp 正在運作。
- 傳送中：按鈕下方顯示簡短文字「正在送出位置」。
- 持續監測：顯示「Yelp 正在整理求助資訊」。

這裡的文字要短，因為使用者可能沒有能力閱讀完整句子。

### 3.3 儀表板入口出現

按下中央按鈕後，畫面底部出現一個新的按鈕：「查看狀態」。

這個按鈕不是一開始就出現，原因是第一秒最重要的事情只有啟動求助。當事件已建立後，才提供查看後台狀態的入口。

「查看狀態」可以讓使用者、家屬或現場協助者知道 Yelp 正在做什麼，而不是讓人感覺按下按鈕後毫無回應。

## 4. 狀態儀表板

### 4.1 儀表板目的

儀表板不是一般的分析後台，而是「求助資料流的可視化狀態」。它要回答三個問題：

- Yelp 已經知道什麼？
- 哪些資料已送出？
- 哪些資訊已被接收或需要補強？

儀表板應以手機底部抽屜或獨立頁面呈現。第一版建議使用底部抽屜，因為它不會讓使用者完全離開求助主畫面。

### 4.2 儀表板區塊

儀表板可以分成四個主要區塊。

#### 4.2.1 事件狀態

顯示目前求助事件的整體狀態。

建議欄位：

- 事件 ID
- 建立時間
- 目前狀態
- 最後同步時間

狀態範例：

- 已建立：本機已建立求助事件。
- 已送出：資料已送達後端資料庫。
- 已讀：後端或接收端已讀取此事件。
- 需複核：系統判讀信心不足，需要人工確認。

#### 4.2.2 位置狀態

顯示位置資料的建立與送出狀態。

建議狀態：

- 定位中
- 已建立
- 已送出
- 已讀
- 定位失敗

位置可以顯示為簡短文字，例如「已取得 GPS 位置」，也可以提供座標預覽。第一版不一定需要完整地圖，避免畫面變重。

#### 4.2.3 災情情報 JSON 預覽

顯示目前系統整理出的災情情報。

這是 Yelp 的核心，因為它代表混亂訊號已被轉換為可交換的防災資料積木。

預覽不需要一開始顯示完整 JSON，可以先顯示摘要：

- 事件類型
- 嚴重程度
- 恐慌程度
- 信心分數
- 推測需求
- 脆弱處境標記
- 建議交付元件

使用者點開後，才顯示完整 JSON。

範例摘要：

- 災害類型：淹水
- 嚴重程度：4 / 5
- 恐慌程度：0.86
- 信心分數：0.72
- 推測需求：救援、行動協助
- 交付目標：救援派遣、醫療支援、地圖熱區

#### 4.2.4 訊號來源狀態

顯示 Yelp 目前正在使用哪些資料來源。

第一版可以包含：

- 文字訊息
- 語音轉錄
- 語音特徵
- 位置

未來可擴充：

- 手勢線索
- 鍵盤求助訊息
- 積木式回報

每個訊號來源都應有狀態，例如：

- 未啟用
- 收集中
- 已建立
- 已送出
- 已整合進 JSON

## 5. 狀態語言設計

Yelp 的狀態文字要非常清楚，避免工程術語直接暴露給使用者。

建議狀態對照：

- 已建立：手機已成功建立這筆資料。
- 已送出：資料已送到 Yelp 後端。
- 已讀：接收端已讀取這筆資料。
- 判讀中：系統正在整理線索。
- 需複核：目前判斷可能不完整，需要人協助確認。
- 暫存中：網路不穩，資料先保存在手機。
- 等待同步：網路恢復後會自動送出。

這些狀態很重要，因為災害中最可怕的不是只有危險本身，還有「我不知道求助有沒有出去」。Yelp 要讓使用者知道資料正在前進。

## 6. 視覺方向

Yelp 的視覺應該是克制、清楚、有安全感，而不是華麗。

### 6.1 色彩

建議採用高對比但不刺眼的色彩。

- 背景色：`#F7F4EF`，偏暖白，降低冷冰冰感。
- 主要行動色：`#D92D20`，用於中央求助按鈕。
- 成功狀態：`#2E7D32`，表示已送出、已讀。
- 警示狀態：`#B7791F`，表示等待同步、需注意。
- 錯誤狀態：`#B42318`，表示定位失敗或送出失敗。
- 文字主色：`#1F2933`，保留清楚閱讀性。
- 次要文字：`#5D6673`。

深色模式可使用：

- 背景色：`#101418`
- 面板色：`#171D23`
- 文字主色：`#F5F7FA`
- 次要文字：`#B8C0CC`
- 主要行動色仍使用 `#F04438`

### 6.2 字體

手機端建議使用系統字體，確保載入快速與可讀性。

- 中文：系統黑體，例如 Microsoft JhengHei、PingFang TC、Noto Sans TC。
- 英文與數字：系統 Sans Serif。

建議字級：

- 主按鈕文字：24px，700 weight。
- 狀態標題：18px，700 weight。
- 區塊標題：15px，700 weight。
- 內文：14px，400 weight。
- 輔助資訊：12px，400 weight。

## 7. 元件設計

### 7.1 中央求助按鈕

位置：畫面正中央。

形狀：大型圓形按鈕。

尺寸：手機寬度的 42% 到 52%，最小不低於 160px。

狀態：

- 預設：紅色實心，文字為「Yelp」或「發出求助」。
- 按下中：短暫縮小或震動回饋。
- 已啟動：按鈕外圈緩慢脈動。
- 傳送中：按鈕下方顯示狀態。
- 失敗：不取消事件，只顯示「暫存中」或「等待同步」。

### 7.2 查看狀態按鈕

位置：中央按鈕啟動後出現在底部。

用途：開啟狀態儀表板。

文字：`查看狀態`

設計：比中央按鈕低調，避免搶走主要行動。

### 7.3 狀態列項目

每個狀態項目包含：

- 名稱
- 狀態標籤
- 最後更新時間
- 簡短說明

例如：

- 目前位置
- 已送出
- 06:42:18
- GPS 位置已送達後端

### 7.4 JSON 預覽卡

JSON 預覽卡分為摘要與詳細兩層。

摘要層給人讀：

- 災害類型
- 嚴重程度
- 恐慌程度
- 信心分數

詳細層給系統與技術協作者讀：

- 完整 JSON
- 欄位定義
- 判斷依據

## 8. 版面架構

### 8.1 主畫面

手機主畫面由上到下：

1. 頂部小字：Yelp
2. 中央大按鈕
3. 一行短狀態文字
4. 啟動後出現的「查看狀態」按鈕

主畫面不放列表、不放地圖、不放設定入口。設定可以之後再加，但第一版不該干擾核心流程。

### 8.2 儀表板抽屜

底部抽屜由上到下：

1. 事件總狀態
2. 位置狀態
3. 災情情報摘要
4. 訊號來源狀態
5. 完整 JSON 展開按鈕

抽屜應可關閉，回到主畫面。

## 9. MVP 畫面範圍

第一版建議只做兩個主要畫面：

- 求助主畫面
- 狀態儀表板

不要一開始做完整帳號系統、歷史紀錄、地圖總覽或設定中心。Yelp 的第一版要證明的是：使用者按一下，系統就開始把混亂訊號轉成可交換的災情資料。

## 10. 與防災積木的關係

Yelp 的手機介面只是入口，真正的交付是後台持續生成的災情 JSON。

主畫面負責降低求助負擔。

儀表板負責讓資料流可見。

JSON 負責讓後續防災積木可接續處理。

因此 Yelp 的 UX 不是單純追求漂亮，而是支撐整個積木式架構：使用者只做最少動作，系統把狀態、位置、訊號與判讀結果整理成可以被其他元件堆疊的資料。
