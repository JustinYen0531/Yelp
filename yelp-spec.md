# Yelp Spec

> 本文件為技術規格。專案的「為什麼」與定位請見 [yelp-positioning.md](yelp-positioning.md)。
> AI Agent 的輸出 JSON 規範請見 [yelp-agent-spec.md](yelp-agent-spec.md)。

## 1. 理念

### 1.1 對個人：承認混亂

Yelp 由兩個詞組成：yell 與 help。

它代表的不是一個冷冰冰的求救按鈕，而是一種更貼近人類真實狀態的防災元件：人在災難中可能尖叫、語無倫次、重複同一句話、停頓、哭喊、打錯字，甚至無法完整描述自己發生了什麼。Yelp 的出發點是承認這件事。

我們不該檢討受害者，要求他在極端壓力下保持冷靜、正確填表、完整描述現場。真正需要被要求進步的是系統本身。系統應該學會辨識混亂訊號，從不完整、非結構化、恐慌化的輸入中，盡可能提取對救援有用的災情情報。

Yelp 的核心信念是：以最低的使用者負擔，換取最多可用的危機資訊。

### 1.2 對群體：把混亂當訊號

但 Yelp 不只是一個面對「單一受災者」的工具。
更關鍵的是：當許多人同時陷入混亂時，**群體的混亂本身就是一種訊號**。

一個人按 yelp，是弱訊號。
一百個人在同一區域同時按 yelp，是災情輪廓。

Yelp 因此有兩層身分：

- **個人層**：把單一使用者的混亂語句轉成結構化災情 JSON。
- **群體層**：把大量結構化 JSON 聚合、去重、排序、畫熱區，協助指揮中心提前感知尚未被任何人完整通報的災害事件。

這個雙層性是 Yelp 與一般「災情通報 app」最重要的差別。

### 1.3 對既有系統：增強而非取代

Yelp **不取代 119**。
119 該打的時候必須打，因為它是同步通話、人對人對話、即時派遣。

Yelp 補上 119 接不到的環節：

- 通話失敗仍能留下定位與恐慌訊號
- 同一個事件被多人同時感知時可以聚合
- 不會說、不能說、來不及說的人也能貢獻一個訊號片段

換句話說，Yelp 是 **119 的資訊增強層**，不是它的對手。

## 2. 動機

### 2.1 現有工具的盲點

現有的災情通報工具通常假設使用者仍有能力清楚輸入資訊，例如選擇災害類型、填寫地點、描述狀況、確認需求。然而災害現場往往不是這樣。

一個真正需要求助的人，可能正在逃跑、受困、哭泣、顫抖、照顧家人，或在斷續訊號與高度恐懼中嘗試發出訊息。這時候，每多一個步驟，都可能讓通報中斷。每要求一次「請完整描述」，都可能錯過幾秒鐘的搶救時間。

### 2.2 不只是 UX 問題，是資料模式問題

更深層的問題是：傳統通報系統（包含 119）本質上是「強訊號、低樣本」的模式 —— 只有**會打、打得通、講得清**的人，才會進入系統。
這個模式對最需要被救的人最不公平：

- 太恐慌、被困、無法講話、聽障、外語、孩童、長輩、網路不穩、快失去意識、不知道自己在哪 ——
  這些人在「請冷靜描述現況」的門檻前就被擋住了。

Yelp 想開的是相反的模式：**弱訊號、高樣本**。
不要求單一訊號完整、清楚、準確，而是接受大量片段化、模糊、恐慌化的訊號，靠 AI 前處理與群體聚合補回完整性。

### 2.3 解決方向

Yelp 想解決的不是「怎麼讓人填得更完整」，而是兩個更基本的問題：

1. **個人層**：「當人無法完整表達時，系統如何仍然理解他？」
2. **群體層**：「當無法被任何單一個人完整描述的災害正在發生時，系統如何從群體訊號中拼出輪廓？」

因此，本專案第一階段主打的 MVP 是：

**混亂訊息 / 恐慌語言 → 結構化災情情報 JSON**

使用者可以輸入語音、文字或混亂片段，Yelp 嘗試從中判讀事件類型、位置線索、嚴重程度、恐慌程度、可信度與可用欄位，最後輸出能被其他防災系統、地圖系統、指揮中心或 AI agent 讀取的標準化資料。

這份 JSON 是「個人層」的核心交付物；同時它也是「群體層」聚合分析的最小單位 —— 一旦標準化，後續就可以做去重、排序、熱區形成、優先派遣排序等所有群體層動作。MVP 雖然先聚焦個人層，但 schema 從一開始就是為群體層設計的。

## 3. 問題分析

### 3.1 災害中的人不一定能正常描述

災害現場的資訊並不會以乾淨的表格出現。人可能只說出「水一直進來」、「我媽還在裡面」、「不知道在哪裡」、「好晃」、「救命」、「拜託快點」這類片段。這些話看似混亂，卻包含重要線索。

例如：

- 「水一直進來」可能指向淹水、豪雨、地下室受困。
- 「好晃」可能指向地震或建物受損。
- 「我媽還在裡面」可能代表有人員受困。
- 連續重複「快點、快點」可能代表高恐慌或時間急迫性。

Yelp 的目標不是把這些訊息清洗成漂亮文字，而是保留其中的危機價值。

### 3.2 傳統通報流程對高壓情境太重

傳統通報常需要使用者主動完成多個動作：

- 選擇災害類型
- 輸入地址
- 描述災情
- 填寫人數或需求
- 送出表單

這些步驟在平時合理，但在災害當下可能太重。Yelp 的設計方向是把理解責任轉移到系統端，讓使用者只需要自然發出訊號，系統在背景完成辨識、整理與轉換。

### 3.3 語音不只是文字內容

如果只是把語音轉成文字，會失去許多重要訊號。人在恐慌時的音高、語速、停頓、顫抖、呼吸急促、音量變化，都可能透露壓力程度與危急程度。

Yelp 主張語音應被視為多層訊號，而不是單純的文字來源：

- 說了什麼
- 怎麼說
- 說得多急
- 是否停頓或中斷
- 是否重複
- 是否出現顫抖或高壓反應

這些訊號可以協助系統判斷事件嚴重程度與通報優先順序。

### 3.4 災情資料需要能被交換

單一 App 的畫面不是 Yelp 的終點。真正有價值的是把混亂訊號轉成可交換、可串接、可被後續系統利用的災情情報。

災害現場不是「一個人看懂就好」的情境，而是多個單位與多種系統要同時接手。消防、警察、醫療院所、避難所、里長、志工團隊、物資平台、地圖系統、簡訊廣播、儀表板與 AI agent，關心的欄位並不相同。

同一則混亂訊息：

「救命，水一直進來，我爸不能走，拜託快點」

對不同元件代表不同任務：

- 消防元件需要知道是否需要立即救援。
- 醫療元件需要知道是否有高齡或行動不便者。
- 避難所元件需要知道是否需要無障礙空間。
- 地圖元件需要知道是否出現淹水熱點。
- 物資元件需要知道是否可能需要輪椅、擔架或車輛。
- AI agent 需要知道還缺哪些關鍵資訊，才能追問或標記人工複核。

因此，JSON 的目的不是炫技，也不是單純讓資料看起來整齊，而是讓一則混亂求救可以被拆成許多防災積木都接得住的欄位。Yelp 在整個防災流程中的定位，是把「人類崩潰時的訊號」轉成「公共系統可交換的求救資料」。

欄位化 JSON 的價值有三個：

- 可讀：人類可以直接理解欄位與判斷依據。
- 可接：API、資料庫、地圖、儀表板、MCP server 與其他元件都可以讀取。
- 可累積：多筆災情可以被統計、去重、排序、分流、畫熱區。

換句話說，Yelp 的 JSON 是災情情報積木的共同接口。它讓後續元件不需要重新理解混亂語句，而是可以直接接續處理自己負責的任務。

因此 Yelp 的輸出應採用欄位化 JSON，例如：

```json
{
  "event_type": "flood",
  "severity": 4,
  "panic_level": 0.83,
  "confidence": 0.71,
  "location": {
    "lat": 23.4,
    "lng": 121.3
  },
  "signals": {
    "voice": true,
    "text": true,
    "gesture": false
  },
  "exchange_targets": [
    "rescue_dispatch",
    "medical_support",
    "shelter_accessibility",
    "flood_hotspot_map"
  ],
  "recommended_actions": [
    "dispatch_rescue_team",
    "prepare_mobility_assistance",
    "mark_as_high_priority"
  ],
  "source_evidence": [
    "speaker repeatedly mentioned water entering the room",
    "speech rate increased and pauses became shorter",
    "message contained repeated urgent phrases"
  ]
}
```

這樣的資料可以被 API、MCP server、GIS 系統、災害儀表板、志工協作平台或指揮調度系統讀取。

## 4. 解決方案

### 4.1 核心元件：Panic Signal Interpreter

Yelp 第一階段的核心元件是 Panic Signal Interpreter，也就是「恐慌訊號判讀元件」。

它的任務是接收非結構化輸入，包含語音轉錄文字、原始語音特徵、使用者打出的混亂文字，並輸出標準化災情 JSON。

核心流程：

1. 接收輸入：語音、文字、位置或其他背景訊號。
2. 擷取線索：分析災害關鍵詞、情緒強度、重複語句、語速停頓等特徵。
3. 推測事件：判斷可能災害類型，例如地震、淹水、火災、受困、醫療需求。
4. 評估嚴重程度：根據危險詞、人員受困、語音壓力、訊息急迫性推估 severity。
5. 評估恐慌程度：根據語音與語言特徵推估 panic_level。
6. 評估可信度：根據訊號一致性、資料完整度與來源數量給出 confidence。
7. 輸出 JSON：提供其他系統交換與使用。

### 4.2 MVP 範圍

第一版不追求完整 App，而是聚焦在最能展現專案價值的核心能力。

MVP 建議包含：

- 一個簡單輸入介面，可輸入混亂文字或語音轉錄結果。
- 一組模擬語音特徵欄位，例如語速、停頓、音高變化、顫抖程度。
- 一個 AI 或規則混合判讀流程，將訊息轉成災情 JSON。
- 一份 JSON Schema，定義輸入與輸出格式。
- 一組 client sample code，展示其他系統如何呼叫此元件。
- 幾組災害情境範例，展示同樣混亂的求助語句如何被轉為欄位化資訊。

暫緩到第二階段的功能：

- 單鍵定位求助
- 手勢語意輸入
- 背景鍵盤訊號整合
- 可視化積木回報介面
- 完整地圖或指揮中心儀表板

這些功能仍是 Yelp 的完整願景，但不應在第一版同時實作，否則專案會失焦。

### 4.3 預期輸出欄位

基礎輸出至少包含：

- `event_type`: 推測災害類型
- `severity`: 嚴重程度，建議 1 到 5
- `panic_level`: 恐慌程度，建議 0 到 1
- `confidence`: 系統判讀可信度，建議 0 到 1
- `location`: 位置資訊，可含 GPS 或文字地點
- `needs`: 推測需求，例如救援、醫療、撤離、物資
- `people_involved`: 可能受影響人數或受困人員線索
- `vulnerability_flags`: 脆弱處境標記，例如高齡者、兒童、行動不便、受困、需要醫療協助
- `exchange_targets`: 建議交付給哪些後續防災元件，例如救援派遣、醫療支援、避難所、地圖熱區、物資調度
- `recommended_actions`: 系統建議的下一步處理，例如高優先派遣、人工複核、追問缺失資訊、標記熱區
- `source_evidence`: 系統做出判斷的依據
- `raw_input_ref`: 原始輸入參照，供後續稽核或人工複核

### 4.4 積木式資料交付設計

Yelp 的核心交付物不是單一畫面，而是一個可以被其他積木接續使用的災情資料物件。這個資料物件應該盡量穩定、清楚、可擴充，讓不同團隊或不同系統可以只取用自己需要的欄位。

以比賽語言來說，Yelp 是「前處理積木」與「語意轉換積木」：它負責把語音、文字、恐慌特徵等混亂輸入，轉換成標準化欄位；後續的救援派遣積木、地圖視覺化積木、避難所管理積木、物資調度積木、AI 追問積木，都可以在這份 JSON 上繼續堆疊。

設計原則：

- 每個欄位都應有清楚定義，避免不同系統自行猜測。
- 每個判斷都應保留 `confidence` 與 `source_evidence`，方便人工複核。
- 每個後續任務都盡量用 `exchange_targets` 與 `recommended_actions` 標記，讓資料可以自動分流。
- 欄位允許逐步擴充，但基礎欄位應維持穩定，讓其他積木不會因版本變動而失效。
- 輸出格式應能轉接 API、MCP server、GIS 系統、資料庫與儀表板。

如果以更抽象的方式理解，Yelp 想建立的是一種災情情報交換格式；但在本競賽中，我們會以「可堆疊的防災資料積木」來描述它，因為重點不是格式本身，而是讓更多救災功能可以在同一份資料上接力。

## 5. 可行性評估

### 5.1 技術可行性

Yelp 的第一階段不需要立即完成高精度語音模型訓練，可以用可驗證的 MVP 方式逐步實作。

可行做法：

- 語音先以既有 Speech-to-Text 工具轉錄。
- 語音特徵可先用模擬資料或簡化欄位表示。
- NLP 判讀可採用 LLM、規則引擎或混合式方法。
- 災害類型與需求分類可先限定少數類別。
- JSON Schema 可先定義穩定欄位，後續再擴充。

這樣可以讓作品先展示核心資料流，而不必一開始就完成完整聲學模型。

### 5.2 研究可行性

本專案可結合心理學、語言學與危機溝通研究，分析人在高壓情境下的語言特徵，例如重複、破碎句、停頓、語速上升、語意省略、情緒詞增加等。

在比賽階段，可以先清楚說明：

- 目前哪些訊號是真實分析
- 哪些訊號是模擬欄位
- 哪些判斷需要人工複核
- 哪些結果只作為輔助決策，不取代救災人員判斷

這樣可以避免過度宣稱，也讓專案更可信。

### 5.3 競賽可行性

Yelp 符合「積木式設計」精神，因為它不是一個封閉系統，而是一個可被其他防災系統呼叫的語意轉換元件。

它可以作為：

- API 服務型元件
- MCP server
- 事件處理型元件
- 防災資料前處理模組
- 災情通報平台的 AI 判讀外掛

作品不需要完整上線，只要展示核心功能可運作、輸入輸出格式清楚、資料流能被驗證，就符合本競賽對 MVP 的期待。

## 6. 社會影響力

Yelp 的社會價值不只在於技術，而在於它改變了防災系統看待人的方式。

它承認人在災難下會失能，承認恐慌、混亂與不完整都是真實的人類反應。系統不應把這些反應視為雜訊，而應把它們視為需要被理解的訊號。

如果 Yelp 成功，它可以幫助：

- 讓無法完整表達的人仍能被系統理解。
- 降低民眾通報災情的心理與操作負擔。
- 協助救災單位更早取得事件類型、嚴重程度與需求線索。
- 將語音、文字與情緒訊號轉為可交換的公共資料。
- 讓公民科技不只追求效率，也承擔對脆弱處境的理解責任。

Yelp 想傳達的不是「請你冷靜回報」，而是：

就算你無法冷靜，系統也會努力理解你。

這是 Yelp 作為防災元件最重要的公共價值。
